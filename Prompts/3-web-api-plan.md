# Config Service ‚Äî Implementation Plan

A centralized configuration management REST API that lets applications fetch key-value config data at runtime. Built with FastAPI + raw psycopg2 on PostgreSQL, managed with `uv` and a Windows `Makefile`.

## Architectural Patterns

### Layered Architecture (3 Layers)

```
Router  ‚Üí  Service  ‚Üí  Repository  ‚Üí  PostgreSQL
```

| Layer | File | Responsibility |
|---|---|---|
| **Router** | `router.py` | HTTP concerns only ‚Äî parse request, call service, return response |
| **Service** | `service.py` | Business rules, validation, orchestration |
| **Repository** | `repository.py` | All SQL queries; returns plain Python dicts/dataclasses |

Each layer depends only on the layer directly below it. The router never touches the DB directly.

### Repository Pattern
- Each domain (`applications`, `configurations`) has its own repository class.
- Methods are named after their intent: `create()`, `update()`, `get_by_id()`, `list_all()`.
- All SQL is written as explicit parameterized queries (no string concatenation).

### Error Handling Strategy
- Repository raises domain-specific exceptions (e.g. `NotFoundError`, `DuplicateNameError`).
- Service re-raises or maps to HTTP-friendly exceptions.
- A global FastAPI exception handler in `main.py` maps these to JSON error responses (`{"detail": "..."}`) with appropriate HTTP status codes (404, 409, 422, 500).

### ULID Primary Keys
- ULIDs are generated in Python (not the DB) via `python-ulid` at creation time.
- Stored as `VARCHAR(26)` in PostgreSQL.

## Database Setup

### Connection Management (`app/database.py`)
- Use `psycopg2.pool.ThreadedConnectionPool` to manage a pool of connections (min=1, max=10).
- Expose a `get_connection()` context manager that acquires a connection from the pool and returns it on exit.
- All repository methods accept a connection object injected by the service layer ‚Äî this keeps repositories testable.

```python
# Conceptual usage in a repository method:
with get_connection() as conn:
    with conn.cursor() as cur:
        cur.execute("SELECT id, name FROM applications WHERE id = %s", (app_id,))
        row = cur.fetchone()
```

### Configuration (`app/config.py`)
- `pydantic-settings` reads `DATABASE_URL`, `APP_HOST`, `APP_PORT` from `.env`.
- A single `Settings` instance is created at import time and shared app-wide.

### Migrations (`db/migrations/`)
- Plain `.sql` files numbered sequentially (`001_`, `002_`, ‚Ä¶).
- `db/run_migrations.py` connects to the DB, creates a `_migrations` tracking table if absent, then runs any un-applied `.sql` files in order.
- Run via: `make migrate`

**Schema (raw SQL):**
```sql
-- 001_create_applications.sql
CREATE TABLE IF NOT EXISTS applications (
    id          VARCHAR(26) PRIMARY KEY,
    name        VARCHAR(255) NOT NULL UNIQUE,
    comments    TEXT
);

-- 002_create_configurations.sql
CREATE TABLE IF NOT EXISTS configurations (
    id              VARCHAR(26) PRIMARY KEY,
    application_id  VARCHAR(26) NOT NULL REFERENCES applications(id),
    name            VARCHAR(255) NOT NULL,
    comments        TEXT,
    config          JSONB NOT NULL DEFAULT '{}'
);
```

## FastAPI Setup

### App Factory (`app/main.py`)
```python
app = FastAPI(title="Config Service", version="1.0.0")
app.include_router(applications_router, prefix="/api/v1")
app.include_router(configurations_router, prefix="/api/v1")
```
- A `GET /health` endpoint returns `{"status": "ok"}` for liveness checks.
- Global exception handlers are registered here for `NotFoundError` ‚Üí 404, `DuplicateNameError` ‚Üí 409.

### Router Organization
Each domain router uses `APIRouter(prefix="/applications")` / `APIRouter(prefix="/configurations")` so routes read cleanly at the router level.

### Pydantic Models (`models.py` per domain)

**Applications:**
```python
class ApplicationCreate(BaseModel):  # POST body
    name: str
    comments: str | None = None

class ApplicationUpdate(BaseModel):  # PUT body
    name: str | None = None
    comments: str | None = None

class ApplicationResponse(BaseModel):  # Response
    id: str
    name: str
    comments: str | None
```

**Configurations:**
```python
class ConfigurationCreate(BaseModel):
    application_id: str
    name: str
    comments: str | None = None
    config: dict[str, Any] = {}

class ConfigurationUpdate(BaseModel):
    name: str | None = None
    comments: str | None = None
    config: dict[str, Any] | None = None

class ConfigurationResponse(BaseModel):
    id: str
    application_id: str
    name: str
    comments: str | None
    config: dict[str, Any]
```

### Middleware
None planned at this time (authentication is a future feature). A basic CORS middleware can be added later without touching core logic.

## Testing Strategy

- **Co-location rule**: Every `*.py` file has a sibling `*_test.py` in the same folder.
- **Coverage target**: 80% across all core modules.
- **Test client**: `httpx.AsyncClient` with FastAPI's `ASGITransport` for router tests (no live server needed).
- **DB isolation**: Router/service integration tests use a separate test database (`TEST_DATABASE_URL` env var). Each test wraps DB operations in a transaction that is rolled back after the test.
- **Repository unit tests**: Use a real test DB connection (fast, since PostgreSQL is local).
- **Service unit tests**: Mock the repository layer to test business logic in isolation.

### Running Tests
```bash
# All tests
make test

# With coverage report
make test-cov

# Single module
uv run pytest app/applications/router_test.py -v
```
